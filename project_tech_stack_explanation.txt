1. PROJECT OVERVIEW & ARCHITECTURE
--------------------------------------------------------------------------------

The AI Teaching Assistant is a Retrieval-Augmented Generation (RAG) platform 
designed to help students interact with academic materials. It combines 
Computer Vision (OCR/Attention Monitoring) with Large Language Models (Gemini) 
to provide a holistic learning experience.

Backend Architecture:
- Framework: FastAPI (Python)
- Architecture Type: Modular Service Layer
- Key Layers: 
    * API Layer (main.py)
    * Service Layer (llm_manager, attention_tracker)
    * Utility Layer (ocr_engine, document_processor, indexer)
    * Persistence Layer (ChromaDB + JSON)

Frontend Architecture:
- Stack: HTML5, CSS3, Vanilla JavaScript
- Pattern: Single Page Application (SPA) style with Fetch API communication
- Features: Real-time video processing (via OpenCV backend), Markdown rendering, 
  MathJax support for equations.

3. AI / MACHINE LEARNING & RAG STACK
--------------------------------------------------------------------------------

Technology Name: Google Gemini API (via LangChain)
-------------------------------------------------
- Type: Large Language Model (LLM)
- Used In: backend/llm_manager.py
- Purpose: Core "brain" of the project. Generates explanations, answers questions, 
  and creates quizzes.
- Why Chosen: Multi-modal capabilities, large context window, and high-quality 
  reasoning compared to smaller models.
- How It Works: Receives a "Prompt" (context + question) and predicts the next 
  sequence of tokens to form a response.
- Project Usage: Used in "Chat" for answering questions and "Quiz" for 
  generating MCQ/Short-answer questions based on PDF text.

Technology Name: Sentence Transformers (all-MiniLM-L6-v2)
--------------------------------------------------------
- Type: Embedding Model
- Used In: backend/llm_manager.py
- Purpose: Converts text chunks into mathematical vectors (embeddings).
- Why Chosen: Lightweight, fast, and excellent for semantic similarity search.
- Problem it solves: Computers can't compare "text" directly; they compare 
  numbers. This model maps similar meanings to similar numbers.
- Alternatives: OpenAI Embeddings, Cohere.

Technology Name: LangChain
--------------------------
- Type: Orchestration Framework
- Used In: Throughout backend/utils/
- Purpose: Connects LLMs with external data (PDFs) and tools (ChromaDB).
- Why Chosen: Standardizes the "RAG" pipeline.
- Integration Flow: Document -> Text Splitter -> Embeddings -> ChromaDB -> 
  Retriever -> LLM.

Technology Name: RecursiveCharacterTextSplitter
-----------------------------------------------
- Type: Text Chunking Module
- Used In: backend/utils/document_processor.py
- Purpose: Breaks long PDF text into smaller, overlapping "chunks" (800 chars).
- Why Chosen: Maintains semantic context by not cutting sentences mid-way and 
  using overlap to preserve continuity between chunks.

6. ATTENTION MONITORING SYSTEM
--------------------------------------------------------------------------------

The Attention Tracker is a specialized CV module:
1. Webcam Access: Initiated via `cv2.VideoCapture(0)`.
2. Face Detection: Uses `Haar Cascade Frontal Face` model.
3. Logic: 
    * Finds the center of the detected face.
    * Compares it to the screen's center point.
    * If the face "drifts" too far (threshold 0.2), user is marked "Distracted".
4. Scoring: (Attentive Frames / Total Frames) * 100.
5. Files: backend/attention/attention_tracker.py.

8. FULL PROJECT FLOW (THE PIPELINE)
--------------------------------------------------------------------------------

STEP 1: DATA INGESTION
User uploads PDF -> Document Processor detects if Native or Scanned -> 
OCR Engine (if scanned) or PyPDF (if native) extracts text -> 
Text Splitter creates chunks -> Embeddings Model creates vectors -> 
ChromaDB stores vectors + metadata.

STEP 2: QUERY / RAG
User asks question -> LLM Manager embeds question -> ChromaDB retrieves 
relevant context -> Prompt Mode wraps context with "System Instructions" -> 
Gemini generates grounded response.

STEP 3: ENGAGEMENT
User starts Attention Mode -> OpenCV processes webcam -> Real-time score 
displayed -> Result saved to Analytics.

STEP 4: ASSESSMENT
User requests Quiz -> Gemini uses retrieved context to generate MCQs -> 
User submits answers -> Progress Tracker updates mastery score.

9. VIVA PREPARATION: COMMON QUESTIONS & ANSWERS
--------------------------------------------------------------------------------

Q1: Why use RAG instead of just sending the PDF to Gemini?
A: RAG is more cost-effective and handles much larger datasets. It only sends 
the relevant "pieces" of the PDF to the LLM, reducing "token bloat" and 
improving accuracy.

Q2: How do you handle scanned PDFs?
A: We use a hybrid pipeline. We first check the text density with PyPDF. If 
it's low, we use pdf2image to convert pages to PNGs and then run our 
custom OCR Engine (OpenCV + Tesseract).

Q3: What is "Semantic Search"?
A: Unlike keyword search (Ctrl+F), semantic search finds meaning. If you 
search "canine", it can find chunks about "dogs" because their vector 
embeddings are mathematically close in vector space.

Q4: Why ChromaDB and not a SQL database?
A: SQL is for structured data. For AI, we need to perform "Nearest Neighbor" 
searches on vectors, which ChromaDB is specifically optimized for.

Q5: How does the attention tracker work?
A: It's a geometric approach using Computer Vision. We track the face 
bounding box and measure its displacement from the camera's center axis 
to determine if the student is focused on the screen.
